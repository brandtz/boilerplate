# Prompt Lifecycle and Archival Standard

## Purpose

Define a durable, dashboard-friendly lifecycle for prompt session files so that:
- active and completed prompts are clearly separated
- completed work remains discoverable and traceable
- the dashboard can compute progress and display session history without relying on memory
- inserted prompts can be added without renumbering the entire project
- downstream prompts remain synchronized when prerequisites or required reading change

## Core Principles

1. The prompt inventory is the source of truth, not folder contents alone.
2. Prompt files are operational work instructions, not the only project record.
3. A prompt may be archived only after its session handoff has been written and the inventory has been updated.
4. The dashboard must include archived prompts in project totals and completion calculations.
5. Prompt numbering must support insertion without requiring global renumbering.

## Required Prompt Folder Structure

```text
prompts/
  README.md
  index.md
  active/
  archive/
  templates/
  generated/
  draft/
```

## Folder Meanings

- `prompts/active/`
  - prompts that are ready, queued, blocked, or currently in progress
  - these are the only prompts eligible to appear in the "next prompt up" queue

- `prompts/archive/`
  - prompts with terminal statuses such as `done`, `superseded`, `cancelled`, or `replaced`
  - archived prompts remain part of dashboard metrics and history

- `prompts/templates/`
  - reusable prompt templates and authoring patterns

- `prompts/generated/`
  - prompt sets generated by the Master Agent before triage into active or draft

- `prompts/draft/`
  - incomplete or unapproved prompts not yet eligible for execution

- `prompts/index.md`
  - canonical prompt inventory for the project
  - dashboard must read this file first

## Required Prompt Metadata

Every active or archived prompt file must begin with frontmatter.

```yaml
---
prompt_id: "16.0.2"
title: "Security Review Follow-up for Auth Session Handling"
status: "ready"
phase: "review"
project: "project-manager-dashboard"
epic_id: "E4"
story_id: "E4-S2"
task_ids:
  - "E4-S2-T3"
prompt_type: "review"
sequence_group: "16"
inserted_after: "16.0.1"
insert_reason: "Defect discovered during review"
prerequisites:
  - "16.0.1"
required_reading:
  - "agents/context/project-charter.md"
  - "agents/epics/epic-4.md"
  - "agents/handoffs/2026-04-03-16.0.1.md"
downstream_prompts:
  - "17.0.1"
session_handoff: "agents/handoffs/2026-04-03-16.0.2.md"
created_at: "2026-04-03"
completed_at: null
archived_at: null
supersedes: []
superseded_by: []
owner_role: "Application Security Reviewer"
---
```

## Prompt Statuses

Allowed prompt statuses:
- `draft`
- `ready`
- `in_progress`
- `blocked`
- `done`
- `superseded`
- `cancelled`
- `archived`

## Lifecycle Rules

### Draft
- created but not approved for execution
- lives in `prompts/draft/`

### Ready
- approved and waiting to run
- lives in `prompts/active/`

### In Progress
- currently being executed in an agent session
- lives in `prompts/active/`

### Blocked
- cannot proceed until an upstream dependency or decision is resolved
- lives in `prompts/active/`

### Done
- execution completed successfully
- must have a completed session handoff and updated index entry
- all changes must be committed and pushed to the remote repository before the prompt is considered complete
- the executing agent MUST complete the **Session Close-Out Checklist** in the prompt file before committing (see prompt template)
- the close-out checklist includes: updating prompt frontmatter, `prompts/index.md`, `agents/context/status-dashboard.md`, and `agents/context/decision-log.md`
- this applies equally to local agents and cloud agents — no close-out step may be skipped
- after validation, move to `prompts/archive/`
- when moved, set status to `archived` or keep `done` with `archived_at` populated based on dashboard needs; prefer `done` plus `archived_at`

### Superseded
- replaced by one or more newer prompts
- must list `superseded_by`
- move to `prompts/archive/`

### Cancelled
- no longer intended for execution
- move to `prompts/archive/`

## Archival Rule

A prompt may be moved from `prompts/active/` to `prompts/archive/` only when all of the following are true:
1. The session has ended.
2. A handoff file exists.
3. The handoff file path is recorded in prompt metadata.
4. `prompts/index.md` has been updated.
5. Any affected downstream prompts have had their prerequisites and required reading reviewed.

## Canonical Inventory Rule

`prompts/index.md` is the canonical inventory.

The dashboard must not infer project truth solely from folder contents because:
- completed prompts are archived
- some prompts may be superseded
- some prompts may be draft-only
- some prompts may exist temporarily in generated form

## Required Prompt Index Columns

`prompts/index.md` must maintain a table or structured section with at least:
- prompt_id
- title
- status
- phase
- epic_id
- story_id
- task_ids
- location
- inserted_after
- prerequisites
- downstream_prompts
- session_handoff
- created_at
- completed_at
- archived_at
- next_action_note

## Dashboard Reading Order

The dashboard should use this read order:
1. `prompts/index.md`
2. prompt files in `prompts/active/` and `prompts/archive/`
3. associated handoff files
4. epic/story/task tracking artifacts

## Next Prompt Up Rule

The dashboard should determine the next prompt using `prompts/index.md` and the following priority:
1. status = `ready`
2. prerequisites satisfied
3. not blocked
4. lowest sequence according to numeric prompt order
5. if multiple exist, prefer the oldest created prompt unless manually pinned

## Prompt Numbering Standard

### Base Pattern

Prompt IDs use hierarchical decimal notation:
- `16.0.1`
- `16.0.2`
- `17.0.1`

Interpretation:
- first segment = major sequence group or milestone
- second segment = insertion lane or branch family
- third segment = ordinal prompt within that lane

### Insertion Rule

If a new prompt must be inserted before what would have been the next prompt:
- keep the existing future prompt ID unchanged if possible
- create the inserted prompt in the same major sequence family
- update the inventory and downstream references

Example:
- existing flow: `16.0.1` then `17.0.1`
- new defect/review prompt needed before `17.0.1`
- create `16.0.2`
- update `17.0.1` prerequisites to include `16.0.2` if required
- update required reading to include the `16.0.2` handoff if relevant

### Superseding Rule

If a prompt is replaced rather than inserted:
- mark the original prompt `superseded`
- create the replacement prompt
- record `superseded_by` and `supersedes`
- move the superseded prompt to `prompts/archive/`

## Downstream Synchronization Rule

When a prompt is inserted, replaced, or materially changes scope, the owning agent must review all affected downstream prompts and update:
- prerequisites
- required reading
- stop conditions if dependencies changed
- downstream consumer references
- index entries

## Required Session Handoff

Each executed prompt must create a handoff with at least:
- prompt_id
- title
- execution date
- role used
- summary of work completed
- files created
- files modified
- files removed
- tests run
- validation results
- blockers or open risks
- decisions made
- required downstream updates
- recommended next prompt

## Dashboard Implications

The dashboard must:
- count archived prompts in total project scope
- treat `done` or archived terminal prompts as completed work
- allow click-through from a prompt inventory row to the corresponding handoff
- distinguish active work from historical work
- show when a prompt has been superseded or replaced
- show if downstream prompt metadata may be stale

## Required Review Rule

Any change to prompt sequencing or archival behavior must be reviewed by:
- Master Agent / Orchestrator
- Product or Project Management role

## Mandatory Governance References in Prompts

Every prompt's `required_reading` frontmatter MUST include:
- `agents/standards/prompt-lifecycle-standard.md`
- `agents/standards/handoff-standard.md`

This ensures that any executing agent — whether local (VS Code) or cloud (GitHub Copilot) — has access to the close-out rules and governance expectations regardless of conversational context.

Every prompt body MUST include a `## Session Close-Out Checklist` section (see `agents/templates/prompt-template.md`) that explicitly enumerates all governance updates required before committing. The old `## Commit and Push` section is deprecated and must not be used.
- Repo Knowledge Curator or equivalent governance role

## Implementation Notes

Recommended supporting files:
- `prompts/README.md`
- `prompts/index.md`
- `agents/standards/prompt-lifecycle-standard.md`
- `agents/templates/prompt-session-template.md`
- `agents/templates/session-handoff-template.md`

## Minimum Acceptance Criteria

This standard is considered implemented when:
1. the new prompt folder structure exists
2. `prompts/index.md` exists and is populated
3. active and archive folders are in use
4. prompt metadata is standardized
5. session handoff template exists
6. the dashboard spec and data contract acknowledge archived prompts
7. prompt authoring rules include insertion and downstream synchronization behavior
